# 13.3 — Migrate `backend/routers/markets.py` exceptions to `AppError`

## Scope

Replace every `raise HTTPException(...)` in `backend/routers/markets.py` with `raise AppError(...)` using the `ErrorCode` enum from `observability`. Replace every `f"...: {e}"` interpolation with `original=exc`. **Does not touch** `markets.py::admin_resolve` (lines 470–496) — that handler is deleted in 13.9.

Largest router and the biggest leak surface. Reviewer should diff line-by-line against AUTHZ_MATRIX §7b and the design-doc §2d leak inventory.

## Acceptance criteria

- [ ] Every `raise HTTPException(...)` in `markets.py` is replaced with `raise AppError(<status>, ErrorCode.<NAME>, "<safe message>", ...)`. Status codes match what was there before — the migration is exception-class-only, not status-code changes.
- [ ] Every `f"...: {e}"` interpolation listed in design doc §2d for this file (lines 287, ~330, ~376–388, 441, 496) is gone. The exception is captured to Sentry via `original=exc`; the user message is hardcoded safe text. Use the `bets.py` patterns (lines 50–75, 103–133) as the reference for common database-error → user-message mappings.
- [ ] When the underlying error has identifying context (a market id, a side, a user-supplied amount), pass it via `extras={...}` so Sentry has it for triage.
- [ ] `admin_resolve` handler at lines 470–496 is **untouched** in this PR. It moves entirely (to admin.py) in 13.9.
- [ ] Existing tests pass: `cd backend && pytest tests/`. No test edits expected.
- [ ] No new `from fastapi import HTTPException` imports added to this file. Existing import can stay (one direct use may remain — see Notes), or remove it if the file no longer references `HTTPException`.

## Mapping table (for the migration)

Use these `ErrorCode` mappings as the default. Add a comment in the code if a specific case calls for a different code.

| Old detail string fragment | ErrorCode | Status |
|---|---|---|
| "Market not found" | `MARKET_NOT_FOUND` | 404 |
| "Market is not open" | `MARKET_NOT_OPEN` | 409 |
| "Market is already resolved" | `MARKET_ALREADY_RESOLVED` | 409 |
| "Market close date has passed" | `MARKET_CLOSE_DATE_PASSED` | 409 |
| "Only the market creator" | `RESOLUTION_NOT_CREATOR` | 403 |
| "Resolution window has expired" | `RESOLUTION_WINDOW_EXPIRED` | 409 |
| "Dispute already filed" | `DISPUTE_ALREADY_FILED` | 409 |
| "Bettors cannot dispute" | `DISPUTE_BETTOR_RESTRICTED` | 403 |
| "Creators cannot dispute" | `DISPUTE_CREATOR_RESTRICTED` | 403 |
| "You've already voted" (dispute) | `DISPUTE_VOTE_DUPLICATE` | 409 |
| Generic 5xx wrapping | `INTERNAL_ERROR` | 500 |

When in doubt, prefer a more specific code over a generic one — the frontend's UX hooks off the code.

## Notes

- The `admin_resolve` handler stays untouched. Don't be tempted to migrate it here — 13.9 deletes it entirely as part of Gap B-3 cleanup.
- For wrapping Supabase RPC errors, follow `bets.py`'s pattern: parse the error message text (which is a stable Postgres/RPC error), map to the appropriate `ErrorCode`, then `raise AppError(..., original=exc) from exc`.
- If you discover a leak that needs an `ErrorCode` not yet in the enum, **add it to the enum in `observability.py` in this PR** (small extension is fine; large additions warrant a separate PR).
