# 13.2 — Frontend: extend `ApiError`, add `parseApiError` helper

## Scope

Extend the existing `ApiError` class in `frontend/src/lib/api.ts` to parse the new envelope shape (introduced by 13.1 backend-side), and add a `parseApiError(err, fallback)` helper. **Backwards-compatible**: parses both old plain-string `detail` (today's shape) and the new envelope. No call site changes in this PR — those happen in 13.6.

This PR is independent of 13.1 — they can ship in either order. The frontend just needs both shapes parseable so neither flag day breaks the UI.

## Acceptance criteria

### `frontend/src/lib/api.ts`

- [ ] Exports `ApiErrorBody` interface:
  ```ts
  export interface ApiErrorBody {
    code: string;
    message: string;
    errors?: { field: string; message: string; type: string }[];
  }
  ```
- [ ] `ApiError` class is extended with public readonly fields: `code: string`, `userMessage: string`, `fieldErrors: { field: string; message: string }[]`, `raw: string`.
- [ ] `ApiError` constructor parses `raw` as JSON. If the parsed value has `detail` that's an object with `code` + `message`, populate the new fields from it. Otherwise leave defaults: `code = "UNKNOWN"`, `userMessage = `Request failed (${status}).``, `fieldErrors = []`. **Never throws on malformed input.**
- [ ] Constructor preserves the existing `super(...)` call so `Error.message` is the user-facing message (not the raw body) — this means existing `err.message` references continue to work and surface the right text.
- [ ] Exports `parseApiError(err: unknown, fallback = "Something went wrong.") => { code, message, fieldErrors }` matching features/extension/13-backend-hardening.md §1e:
  - `err instanceof ApiError` → use the parsed fields.
  - `err instanceof Error && err.name === "AbortError"` → `code: "ABORTED"`, message "Request cancelled.".
  - `err instanceof TypeError && /fetch/i.test(err.message)` → `code: "NETWORK_ERROR"`, message "Connection failed. Check your internet.".
  - Otherwise → `code: "UNKNOWN"`, message = `fallback`.

### Backwards compatibility

- [ ] Construct `new ApiError(500, "Internal server error.")` (today's plain-string shape). Confirm: `code === "UNKNOWN"`, `userMessage === "Request failed (500)."`, `message` falls through to `userMessage`.
- [ ] Construct `new ApiError(409, '{"detail": {"code": "BET_INSUFFICIENT_BALANCE", "message": "Not enough."}}')`. Confirm: `code === "BET_INSUFFICIENT_BALANCE"`, `userMessage === "Not enough."`, `message === "Not enough."`.
- [ ] Construct `new ApiError(422, '{"detail": {"code": "VALIDATION_FAILED", "message": "Validation failed.", "errors": [{"field": "title", "message": "required", "type": "missing"}]}}')`. Confirm: `fieldErrors` populates correctly.

### Tests (light unit)

- [ ] Add unit tests in `frontend/src/lib/__tests__/api.test.ts` (or wherever existing tests live) for the three constructor cases above and the four `parseApiError` branches.

## Notes

- Existing call sites that do `err instanceof Error ? err.message : "..."` will **continue to work** thanks to the `super(...)` preservation. They get migrated to `parseApiError` in 13.6.
- The `fieldErrors` field is unused in this PR — it's there so 13.6 (or 17 — market creation validation) can surface per-field validation errors without needing to extend `ApiError` again.
- Keep the file small; avoid pulling in extra deps. JSON parsing is via the platform `JSON.parse`.
