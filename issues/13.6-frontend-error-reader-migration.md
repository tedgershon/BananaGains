# 13.6 — Frontend: sweep `err.message` patterns to `parseApiError`

## Scope

Mechanical sweep across `frontend/src/app/**/*-client.tsx` and any other call sites: replace `err instanceof Error ? err.message : "..."` patterns with `parseApiError(err, "...").message`. For places that benefit from switching on the error code (e.g., 409 → `CLAIM_ALREADY_TODAY`-specific UX), use the `code` field from `parseApiError`'s return value.

Pure mechanical change — no functional differences for the user except that toast text is now the safe envelope `message` rather than (potentially) raw exception text.

## Notes

- For places that want **per-code branching** (like `handleClaim`), use:
  ```tsx
  const { code, message } = parseApiError(err, "Action failed.");
  if (code === "CLAIM_ALREADY_TODAY") {
    setError("Come back tomorrow for another claim.");
  } else if (code === "CLAIM_AT_BALANCE_CAP") {
    setError("You're at the balance cap.");
  } else {
    setError(message);
  }
  ```
- For places that just want the message, use `parseApiError(err, "fallback").message`.
- The `fieldErrors` field is populated for `VALIDATION_FAILED` responses. Forms that want per-field error display should iterate `fieldErrors` and assign each `{field, message}` to the corresponding input. Defer per-form integration to feature 17 if it's not trivial in this PR — this PR's only goal is "no raw `err.message`".
- After this PR, if a backend route hasn't been migrated yet (shouldn't be possible because 13.3–13.5 are dependencies), `parseApiError` falls back gracefully to `code: "UNKNOWN"` + the supplied fallback message. Backwards-compat is the design.
