# 18.2 — Cleanup: act on the inventory

Work spec: see `issues/18.1-inventory-output.md`.

## Scope

Execute the actions specified in `issues/18.1-inventory-output.md`. Each row maps to one of: relocate to `lib/constants.ts`, move to `seeds/`, add documenting comment, or delete.

**This PR's diff size is determined by 18.1.** If 18.1's summary recommends splitting into 18.2a / 18.2b, do that — open separate PRs per file group rather than one giant cleanup.

## Acceptance criteria

### For each `runtime-config` row in the inventory

- [ ] Consolidate into `frontend/src/lib/constants.ts` (create if missing). Common candidates:
  - `CATEGORIES`
  - `TIME_ZONE_OPTIONS`
  - `OPTION_COLORS`
- [ ] All call sites that previously declared the constant inline now import from `@/lib/constants`. **No duplicate definitions remain.**
- [ ] If a constant is genuinely scoped to one file and not reused, leave it (don't move purely for the sake of moving). Document this case in the PR description.

### For each `seed-script` row in the inventory

- [ ] Move to a clearly-named directory (`seeds/` at repo root, or `backend/supabase/seeds/` if backend-scoped). Add a `seeds/README.md` explaining what's in there and how to apply (or that the contents are documentation only).
- [ ] Update any references to the old paths.

### For each `demo-scaffolding` row marked "keep + document"

- [ ] Add a comment block at the top of the relevant function/section, e.g.:
  ```ts
  // DEMO MODE: this branch maintains a frontend-only "demo user" sentinel
  // (see project-specs/AUTHZ_MATRIX.md §2) so signed-out visitors get a
  // working preview of the UI shell. The backend never sees a "demo user" —
  // any actual API call from the demo client is `anon` from the backend's
  // perspective. Do not extend this with backend writes.
  ```
- [ ] Comment is concise (3–6 lines) and references the matrix.

### For each `demo-scaffolding` row marked "remove" or each `dead-code` row

- [ ] Delete the code. Remove unused imports left behind.
- [ ] If the deletion breaks tests, fix the tests in the same PR (the test was probably also dead).

### Quality gates

- [ ] `cd frontend && pnpm tsc --noEmit` — zero type errors.
- [ ] `cd frontend && pnpm lint` — passes (or, if lint surfaces unused-import errors, fix them in this PR).
- [ ] `cd backend && pytest tests/` — passes.
- [ ] `cd frontend && pnpm dev` — app boots; manually visit `/`, `/markets`, `/markets/create`, `/admin/review` and confirm nothing broke (categories still appear in dropdowns; date picker still works; etc.).

- **PR description** — list every action you took, mapped to the inventory rows. Reviewer should be able to read the PR and verify "yes, you did exactly what 18.1 said you'd do, no more, no less."
