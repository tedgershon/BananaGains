# 14.1 — Test infrastructure: `conftest.py`, test users, factories

## Scope

Stand up the FastAPI integration-test scaffolding that all of doc 14 builds on. **First TestClient tests in the codebase** — the existing test files (`test_market_schema_validation.py`, `test_market_state_machine.py`) are pure unit tests. After this PR, future tests can import the fixtures defined here.

No assertions added in this PR — just the fixtures plus one smoke test that exercises them.

## Acceptance criteria

### `backend/tests/conftest.py` (new)

- [ ] `app_client` (session-scoped) — yields a `TestClient(main.app)`. Configures dependency overrides:
  - `get_supabase_client` returns a test-scoped Supabase client (real or stubbed; see Notes for which).
  - `get_current_user` / `get_current_user_optional` are **not** overridden globally — auth is supplied per-test via `as_role`.
- [ ] `as_role(role: str)` (function-scoped) — fixture factory returning either a wrapped `TestClient` with the right `Authorization` header set, or `app_client` itself for `anon`. Supports: `"anon"`, `"user"`, `"admin"`, `"super_admin"`. The token for each role resolves to a stable test profile UUID (see next bullet).
- [ ] Test profiles fixture (session-scoped) — creates four deterministic test users in the test DB:
  - `test-user@cmu.edu` with role `user`, balance 1000.
  - `test-admin@cmu.edu` with role `admin`, balance 1000.
  - `test-super@cmu.edu` with role `super_admin`, balance 1000.
  - Each has a stable UUID known to the fixtures (so `seed_market(creator_role="user")` knows whose `creator_id` to use).
  - Cleaned up at session teardown (or use a transactional fixture that rolls back).
- [ ] `seed_market(status="open", creator_role="user", **fields)` — factory for inserting a market in any matrix-relevant state. Defaults produce a minimal valid market; `**fields` overrides any column. Returns the inserted row.
- [ ] `seed_bet(market, role, side, amount=10)` — factory for placing a bet on `market` as `role`. Used to set up `bettor(M)` overlay states.

### `backend/requirements-dev.txt` (new) or update `requirements.txt`

- [ ] Adds `httpx>=0.27.0`, `pytest-asyncio>=0.23.0`. (`pytest` is presumably already there for the existing tests; verify version >= 8.0.)

### `pyproject.toml` or `pytest.ini`

- [ ] `asyncio_mode = "auto"` so async fixtures work without explicit decorators.
- [ ] `testpaths = ["backend/tests"]`.

### Smoke test

- [ ] `backend/tests/test_smoke_fixtures.py` (new): four trivial tests, one per role, that hit `GET /api/auth/me` (or `GET /` health) and assert the expected status. Proves the fixtures wire up correctly.

## Notes

- **Test DB strategy** — two viable options:
  - **(a) Real Supabase project** scoped to `bananagains-test` with RLS enabled. Closest to production; requires a separate DB and credentials in CI secrets.
  - **(b) In-memory or container Postgres** with the consolidated migration applied. Faster, no external service, but Supabase Auth/RLS quirks aren't exercised.
  - **Recommend (a)** for the matrix tests to be meaningful — RLS bypass is one of the things doc 14 needs to catch. Document the credentials wiring in `backend/README.md`.
- **Auth tokens** — for stability, generate JWTs signed with the test JWT secret and the test profile's UUID as `sub`. Matches what `_resolve_user` expects. Avoid going through Supabase Auth's signup flow in tests (slow, brittle).
- **Cleanup discipline** — `seed_market` and `seed_bet` should record what they inserted; a session-scoped teardown deletes them. Or use a transactional fixture (`pytest-postgresql`-style) that rolls back after each test. Pick one and document the choice.
- This PR introduces **no test assertions about behavior**. It's pure scaffolding. Behavior assertions start in 14.2.
