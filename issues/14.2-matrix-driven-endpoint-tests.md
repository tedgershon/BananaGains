# 14.2 — Matrix-driven endpoint tests + coverage assertion

## Scope

Hand-translate AUTHZ_MATRIX §7 into `MATRIX_CELLS` (a Python list of `MatrixCell` records), parameterize a single test function over the list, and add a coverage assertion that catches any `@router.` declared in code but missing from the matrix.

After this PR, every PR runs ~150 individual cell tests on every commit. CI shows exactly which `(method, path, role)` cell failed.

## Acceptance criteria

### `backend/tests/_matrix_cells.py` (new)

- [ ] `MatrixCell` dataclass with: `method: str`, `path: str`, `role: str`, `expected_status: int`, `expected_code: str | None`, `body: dict | None = None`, `setup: Callable | None = None`.
- [ ] `MATRIX_CELLS: list[MatrixCell]` — exhaustive enumeration of every cell in matrix §7a–§7i. **One row per `(endpoint, role)` pair** that has a documented expected outcome. For state-dependent rows (e.g., `M.status='open'`), include a `setup` callable that uses `seed_market(...)` to create the right state.
- [ ] Number of rows ≈ count of cells in matrix §7. Rough target: 150–200 entries.

### `backend/tests/test_endpoint_matrix.py` (new)

- [ ] Single parameterized test:
  ```python
  @pytest.mark.parametrize("cell", MATRIX_CELLS, ids=lambda c: f"{c.method} {c.path} as {c.role}")
  def test_endpoint_matches_matrix(cell, as_role, seed_market, seed_bet, app_client):
      if cell.setup:
          cell.setup(seed_market, seed_bet)
      response = as_role(cell.role).request(cell.method, cell.path, json=cell.body or {})
      assert response.status_code == cell.expected_status, ...
      if cell.expected_code:
          body = response.json()
          assert body.get("detail", {}).get("code") == cell.expected_code
  ```
- [ ] When `expected_code` is in `USER_FACING_MESSAGES` (the catalog mirror), additionally assert `body["detail"]["message"] == USER_FACING_MESSAGES[cell.expected_code]`.

### `backend/tests/test_route_coverage.py` (new)

- [ ] `test_matrix_cells_cover_every_route()` — walks `app.routes` (filtering to those under `/api`), collects `(method, path)` pairs, asserts every one appears at least once in `MATRIX_CELLS`. Failure message lists missing routes.

### Catalog mirror (`backend/tests/_user_messages.py` or inline)

- [ ] Mirrors doc 13 §4 catalog as a Python dict: `{"UNAUTHORIZED": "You need to sign in.", ...}`. Keep in sync with the design doc — if a new code is added, update this dict.

### CI

- [ ] GitHub Actions (or equivalent) runs `cd backend && pytest tests/` on every PR.
- [ ] Test output (xunit XML or junit format) uploaded as an artifact for failure inspection.

## Notes

- **Hand-translate, don't auto-parse** — the matrix tables in `AUTHZ_MATRIX.md` are markdown; auto-parsing them is fragile. A hand-written `MATRIX_CELLS` list is reviewable diff and easy to update. The "is matrix in sync with code?" check is the coverage test, not auto-parsing.
- **Setup callables stay short** — if a cell needs complex multi-step setup (create market → place bet → wait for status transition), put the helper in `conftest.py` and have the cell's `setup` call it.
- **State-dependent cells** are the main complexity. A cell like "`POST /api/markets/{id}/resolve` as `creator(M)` with `M.status='closed'` returns 200" needs the seed factory to (a) create the market with `creator_id = test-user UUID`, (b) set its status to `closed` directly, (c) provide the path `/api/markets/<that-market-id>/resolve` to the test. Plan the cell schema to hold both the path *template* and any path-param resolution callable.
- **Test runtime** — at ~150–200 tests with TestClient overhead, expect 30–90s. Acceptable on every PR. If it grows past 5 min, parallelize with `pytest-xdist`.
- The first run will likely surface real bugs (cells where the matrix says one thing and the code does another). Fix the code, not the matrix — the matrix is the spec.
