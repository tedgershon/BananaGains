# 17.2 — Frontend: mirror tightened limits, clamp date picker, render per-field errors

## Scope

Update `frontend/src/app/markets/create/page.tsx`:

1. Mirror the new max-length constants from 17.1.
2. Clamp the `close_at` `<input type="datetime-local">` `min` and `max` attributes so the picker UI itself prevents out-of-window dates.
3. Wire the envelope's `errors[]` (exposed as `fieldErrors` by `parseApiError`) into per-input error rendering: red inline text under each invalid field, replacing the current "single error toast" pattern for `VALIDATION_FAILED` responses.

Net user-visible change: the create-market form gains contextual error guidance instead of a vague top-level message.

## Acceptance criteria

### Constants

- [ ] At the top of `frontend/src/app/markets/create/page.tsx`, update:
  ```ts
  const MAX_MARKET_TITLE_LENGTH = 120;
  const MAX_MARKET_DESCRIPTION_LENGTH = 800;
  const MAX_MARKET_RESOLUTION_LENGTH = 800;
  const MAX_MARKET_OFFICIAL_SOURCE_LENGTH = 200;
  const MAX_MARKET_CRITERIA_LENGTH = 400;
  const MAX_MARKET_LINK_LENGTH = 500;
  // unchanged:
  const MAX_MARKET_OPTION_LENGTH = 80;
  ```
- [ ] Add new constants:
  ```ts
  const MIN_CLOSE_AT_HOURS = 1;
  const MAX_CLOSE_AT_DAYS = 180;
  ```
- [ ] All existing `<input maxLength={...}>` and `<textarea maxLength={...}>` attributes pick up the new constants automatically (the file already references the constants — verify after the constant edit).

### Date picker clamping

- [ ] On the `<input type="datetime-local">` for `close_at`:
  - Compute `min` = "now + 1 hour" formatted as `YYYY-MM-DDTHH:mm`.
  - Compute `max` = "now + 180 days" formatted the same way.
  - Apply both as JSX `min` / `max` props.
  - Recompute on form mount; don't recompute on every render (use `useMemo` keyed off mount or current time bucket).

### Per-field error rendering

- [ ] Add `errorsByField` state:
  ```tsx
  const [errorsByField, setErrorsByField] = useState<Record<string, string>>({});
  const [submitError, setSubmitError] = useState<string | null>(null);
  ```
- [ ] In the submit-handler `catch` block, replace the current `err instanceof Error ? err.message : "..."` pattern (already mostly migrated by 13.6 — confirm) with:
  ```tsx
  const { code, message, fieldErrors } = parseApiError(err, "Failed to create market.");
  if (code === "VALIDATION_FAILED" && fieldErrors.length > 0) {
    setErrorsByField(Object.fromEntries(fieldErrors.map(e => [e.field, e.message])));
    setSubmitError(null);
  } else {
    setErrorsByField({});
    setSubmitError(message);
  }
  ```
- [ ] Below each form input (Title, Description, Resolution Criteria, Close Date, Category, Link, Official Source, Yes/No/Ambiguity criteria, Options[i]), render the per-field error if present:
  ```tsx
  {errorsByField.title && (
    <p className="text-xs text-danger mt-1">{errorsByField.title}</p>
  )}
  ```
- [ ] When the user edits an input, clear that input's error from `errorsByField` (so they get instant feedback when typing the fix). Other field errors remain visible.

### Submit-error fallback

- [ ] If `submitError` is set (non-validation error path), render it at the top of the form as before — but only when no `errorsByField` entries exist.

## Notes

- **Date formatting for `min`/`max`** — `<input type="datetime-local">` expects `YYYY-MM-DDTHH:mm` (no timezone). Use a small helper to build it from a `Date`. Watch out for timezone drift between user-local and UTC; the existing file has `parseDateTimeLocal` and `zonedLocalDateTimeToUtcDate` helpers — reuse the inverse.
- **Multichoice option errors** — `fieldErrors[].field` for option errors will look like `options.0.label`, `options.2.label` etc. (Pydantic's `loc` joined with `.`). Render those next to the corresponding option input. Use a regex / split helper.
- **Field error vs maxLength** — the input's `maxLength` already prevents typing past the limit. Field errors fire when the *server* rejects (which can happen if the user pastes past the limit, or if the limit changes server-side without a frontend redeploy). Both layers needed.
- **Don't suppress the field error after a user edit** — that masks the cause. Only clear the specific field's error when the user starts editing that field.
- After this lands, doc 15.4's `validation.spec.ts` placeholder should be filled in.
