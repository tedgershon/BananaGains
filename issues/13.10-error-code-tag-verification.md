# 13.10 — Verify every `AppError` raise tags Sentry events with `error_code`

## Scope

End-of-phase verification PR. Confirm that every `ErrorCode` enum value has been exercised at least once in production-equivalent testing and that the corresponding Sentry event carries the `error_code` tag (which `AppError.__init__` from 13.1 attaches via `scope.set_tag("error_code", code.value)`).

**Most likely a no-op code-wise** — if 13.1's `AppError` implementation is correct and 13.3–13.5 migrated routers properly, the tag is already attached on every captured event. This PR's value is the *verification*, not new code.

## Acceptance criteria

### Tag verification

- [ ] Manually trigger at least one error per `ErrorCode` value (some — like `INTERNAL_ERROR` — may require deliberate misuse). For each, capture the corresponding Sentry event and confirm:
  - `tags.error_code` matches the enum value.
  - `user.id` is set (when the error came from an authenticated route).
  - `request_id` is in `extra` (set by `RequestContextMiddleware` from doc 12).
- [ ] Document the event URL or screenshot in the PR description, grouped by `ErrorCode`.

### `rg` checks

- [ ] `rg "raise HTTPException" backend/routers/` returns **zero matches** — `AppError` everywhere now.
- [ ] `rg 'f".*: \{[a-z_]*\}"' backend/routers/` returns **zero matches** (no leaked exception text).
- [ ] `rg "current_user.get\\(\"is_admin\"\\)" backend/routers/` returns **zero matches** (replaced by `require_admin` dependency).
- [ ] `rg "err instanceof Error \\? err.message" frontend/src` returns **zero matches**.

### Optional: smoke test for "forgot to add an `error_code` tag" cases

- [ ] If during verification you find an event WITHOUT the `error_code` tag, audit how the exception was raised. Most likely cause: a code path is still raising plain `HTTPException` directly. Find it and convert to `AppError`. Repeat until clean.

## Notes

- This PR is the audit gate before doc 14 (API contract tests) starts. Doc 14 reads `error_code` from response envelopes; if a response doesn't have one, doc 14's tests can't run.
- If the verification surfaces a new `ErrorCode` that needed to be added but wasn't, add it to the enum in this PR (small extension is fine here since this PR's job is "make sure everything's tagged correctly").
- If you find an `HTTPException` raise that legitimately should *not* go through `AppError` (e.g., something raised by a middleware before the router), document it explicitly in the PR description and consider whether the global `unhandled_exception_handler` from 13.1 is correctly catching it.
- After this PR closes, the matrix §8 should have only Gap B-4 remaining (B-1, B-2, B-3 closed by 13.7, 13.8, 13.9).
