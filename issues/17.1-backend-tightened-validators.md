# 17.1 — Backend: tighten Pydantic limits + add `close_at` window

## Scope

Update `backend/schemas/market.py` validators per the design doc's tightening table. Replace the existing `close_at_in_future` validator with `close_at_in_window` enforcing both a lower (`+1h`) and upper (`+180d`) bound. Update `backend/tests/test_market_schema_validation.py` accordingly.

No router changes. No frontend changes (those are in 17.2). Doc 13's `validation_handler` (already in place via 13.1) automatically packs Pydantic errors into the envelope's `errors[]` array — no plumbing work here.

## Acceptance criteria

### `backend/schemas/market.py`

- [ ] Update length constants:
  ```python
  MAX_MARKET_TITLE_LENGTH = 120                  # was 160
  MAX_MARKET_DESCRIPTION_LENGTH = 800            # was 2000
  MAX_MARKET_RESOLUTION_LENGTH = 800             # was 2000
  MAX_MARKET_OFFICIAL_SOURCE_LENGTH = 200        # was 300
  MAX_MARKET_CRITERIA_LENGTH = 400               # was 1000
  MAX_MARKET_LINK_LENGTH = 500                   # was 2048
  ```
- [ ] Add `close_at` window constants:
  ```python
  from datetime import timedelta
  MIN_CLOSE_AT_DELTA = timedelta(hours=1)
  MAX_CLOSE_AT_DELTA = timedelta(days=180)
  ```
- [ ] Replace the existing `close_at_in_future` validator on `CreateMarketRequest` with `close_at_in_window` enforcing both bounds. Same field name (`close_at`), same `@field_validator` decorator.
- [ ] **`ReviewMarketRequest.close_at`** (admin can edit close_at during review): apply the same window check.
- [ ] No changes to: `MAX_MARKET_CATEGORY_LENGTH`, `MAX_MARKET_OPTION_LENGTH`, `MAX_MARKET_REVIEW_NOTES_LENGTH`, options count rules.

### `backend/tests/test_market_schema_validation.py`

- [ ] Update existing tests to use the new limits:
  - `test_create_market_rejects_oversized_title` → 121 chars (was 161).
  - `test_create_market_rejects_oversized_description` → 801 chars (was 2001).
  - Same pattern for resolution criteria, official source, link, optional criteria.
- [ ] Add new tests:
  - `test_create_market_rejects_close_at_too_soon` — `close_at = now + 30 minutes`. Expect `ValidationError`.
  - `test_create_market_rejects_close_at_too_far` — `close_at = now + 200 days`. Expect `ValidationError`.
  - `test_create_market_accepts_close_at_at_lower_bound` — `close_at = now + 2 hours`. Expect success.
  - `test_create_market_accepts_close_at_near_upper_bound` — `close_at = now + 179 days`. Expect success.
  - `test_review_market_rejects_close_at_too_far` — same upper-bound check on `ReviewMarketRequest`.

## Notes

- **Existing markets in the DB** with `close_at` outside the new window are NOT migrated. They remain whatever they are — this PR only constrains future *creations* and *admin edits*. Expected; no migration needed.
- The 6-month upper bound was chosen as a UX call (per design doc). If the team disagrees during review, change `MAX_CLOSE_AT_DELTA` here and update the design doc — it's deliberately one configurable constant.
- The `from datetime import timedelta` import may already be present (used in `close_at_in_future`); just add it to the import line if not.
- Pydantic v2 syntax — `info: ValidationInfo` parameter pattern — is preserved exactly as the existing validators use it.
