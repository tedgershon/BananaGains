# 15.4 — Error toast allow-list (doc 13 §4) + hygiene checks

## Scope

Two test categories that both protect against information leakage in the rendered UI:

1. **Error toast allow-list:** for each `ErrorCode` in the doc 13 §4 catalog with a user-visible toast path, force the error from the UI, capture the visible toast text, assert it matches the catalog exactly, and assert it does NOT contain raw exception markers (SQL, `HTTPException`, `Traceback`, etc.).
2. **Hygiene checks:** no JWT in DOM after sign-in, no `andrew_id` in error toasts or page content for non-owner contexts, no client-side `useEffect` redirects on protected routes.

### Test pattern (per ErrorCode)

```ts
test("BET_INSUFFICIENT_BALANCE → 'Not enough bananas.'", async ({ page, signInAs, seedMarket, setBalance }) => {
  await signInAs("user");
  await setBalance("user", 0);
  const m = await seedMarket({ status: "open", creatorRole: "admin" });

  await page.goto(`/markets/${m.id}`);
  await page.getByLabel("Amount").fill("10");
  await page.getByRole("button", { name: "Place Bet" }).click();

  // Positive: matches catalog exactly
  await expect(page.getByRole("status")).toHaveText("Not enough bananas.");

  // Negative: no raw schema/SQL/exception text
  const status = await page.getByRole("status").textContent();
  expect(status).not.toMatch(/SQL|Traceback|HTTPException|insufficient_balance|RPC|column|constraint/i);
});
```

The negative-assertion regex is the regression guard. **Every test in this file includes both assertions.**

### Hygiene tests

- [ ] **No JWT in DOM after sign-in:**

  test("no JWT leaks into rendered DOM", async ({ page, signInAs }) => {
    await signInAs("user");

## Notes

- **Toast selector** — assumes the app renders error toasts in a `role="status"` element (ARIA live region). If the actual rendering uses a different role/structure, update the selector here.
- **Forcing `INTERNAL_ERROR`** — hard to do from the UI naturally. One option: a dev-only backend endpoint that always raises `ValueError`, used only in this test file, gated on `SENTRY_ENVIRONMENT == "development"`.
- **Catalog drift** — if a doc 13 §4 message string changes, this PR's tests fail until updated. That's the point — the catalog is the contract.
- **`andrew_id` leak detection** — for the strongest version, seed a market created by user A whose `andrew_id` is a known string (e.g., `test-other@cmu.edu`), then visit relevant pages as user B and grep the rendered DOM. If user A's `andrew_id` ever appears outside contexts that legitimately surface other users (leaderboard public usernames are OK; "Proposed Markets" section is NOT), fail.
- This file's tests are slower than 15.2/15.3 because each one triggers a real backend mutation. Acceptable for the value.
