# 12.1 — Sentry SaaS account, projects, environments, alerts

## Scope

One-time manual setup in the Sentry SaaS console (https://sentry.io). No code changes — purely account configuration plus capturing the resulting DSNs into the team's secret store and the repo's `.env.example` files. **Blocks every other observability issue** because they need the DSNs.

This issue includes the Vercel integration install (so source maps upload automatically on every frontend deploy). It does **not** include any code wiring — that lives in 12.2 (backend) and 12.4 (frontend).

## Acceptance criteria

- [ ] Sentry org `bananagains` exists (or the team-chosen equivalent name), region **US**, plan Developer (free tier) or Team if multiple maintainers need access.
- [ ] Two projects exist under the org: `bananagains-backend` (platform: Python / FastAPI) and `bananagains-frontend` (platform: JavaScript / Next.js).
- [ ] Both project DSNs captured. Stored in: (a) the team's secret store, and (b) Vercel project env vars under `SENTRY_DSN` (backend) and `NEXT_PUBLIC_SENTRY_DSN` + `SENTRY_DSN` (frontend), Production scope.
- [ ] Sentry → Settings → Integrations → Vercel → Install: authorize the Vercel team that owns the `bananagains` projects. Confirm `SENTRY_AUTH_TOKEN` is auto-injected into Vercel build env.
- [ ] One alert rule per project: notify on every new issue, scoped to `environment:production` (so dev errors don't page anyone). Backend rule additionally notifies on error rate > 5/min spike.
- [ ] Notification channel: shared email distro to start. Slack integration deferred.
- [ ] `backend/.env.example` updated with `SENTRY_DSN=`, `SENTRY_ENVIRONMENT=development`, `SENTRY_TRACES_SAMPLE_RATE=0.1` placeholders.
- [ ] `frontend/.env.example` (create if missing) updated with `NEXT_PUBLIC_SENTRY_DSN=`, `NEXT_PUBLIC_SENTRY_ENVIRONMENT=development`, `SENTRY_DSN=`, `SENTRY_ENVIRONMENT=development` placeholders.

## Spec / design references

- features/12-observability.md §"Architecture Overview" (key design decisions: one org / two projects / two environments / US region / sample rates).

## Verification

- Sentry web console: both projects visible under the org, DSNs accessible from Settings → Client Keys.
- Vercel project Settings → Environment Variables: `SENTRY_DSN` (backend) and `NEXT_PUBLIC_SENTRY_DSN` + `SENTRY_DSN` (frontend) populated for Production.
- Vercel build log on the next deploy: contains a Sentry source-map upload step (only matters once 12.4 is also deployed; for now confirm the integration is installed).

## Notes

- **Region is hard to change later.** Confirm US is acceptable per the team's data-handling policy before clicking. Switch to EU only if explicitly required.
- The alert rule can be tuned later. Start permissive ("notify on every new issue") so we don't miss anything in the first week, then tighten once we know baseline noise.
- This issue is the only manual-setup item in the observability phase. Everything after this is code.
