# 09.1 — Enable RLS on `badge_definitions`

## Scope

Supabase's database linter currently flags `public.badge_definitions` as a CRITICAL finding:

> *Table `public.badge_definitions` is public, but RLS has not been enabled.*

Migration `044_badge_definitions.sql` (and the matching block in `consolidated_all_migrations.sql`) creates the table and seeds 25 rows but never runs `ALTER TABLE badge_definitions ENABLE ROW LEVEL SECURITY`. The sibling table `user_badges` (`045_user_badges.sql`) does enable RLS — `badge_definitions` was simply missed.

This PR closes the gap with a new forward-only migration. The catalog stays world-readable (callers via the anon key on the rewards page and the leaderboard need to read it, and the rows are non-sensitive seed data), but writes are no longer reachable from PostgREST under any role — only the service role (which bypasses RLS) and seed migrations can mutate.

Out of scope: any audit of other Supabase linter findings; rewriting `consolidated_all_migrations.sql` from scratch. We append the new migration to the consolidated file and update the migration registry, nothing more.

- [ ] Enables RLS on `badge_definitions`:

- [ ] Adds an idempotent public-read policy (matches the `user_badges` "viewable by everyone" pattern):
  ```sql
  DO $$ BEGIN
      IF NOT EXISTS (
          SELECT 1 FROM pg_policies
          WHERE tablename = 'badge_definitions'
            AND policyname = 'Badge definitions are viewable by everyone'
      ) THEN
          CREATE POLICY "Badge definitions are viewable by everyone"
              ON badge_definitions FOR SELECT
              TO public
              USING (true);
      END IF;
  END $$;
  ```
- [ ] **No** `INSERT` / `UPDATE` / `DELETE` policy is added. Mutations remain service-role-only (which bypasses RLS) — matches how the seed data is loaded in `044_badge_definitions.sql`.

## Notes

- **Why `TO public` instead of `TO authenticated`.** The backend calls Supabase with the anon key from `get_supabase_client` (`backend/dependencies.py`) and forwards the user's JWT only for routes that need `auth.uid()`. The rewards listing is fine either way today, but `TO public` is consistent with `055_fix_notifications_rls.sql` and the seed/test paths that hit the anon role directly. Catalog rows are non-sensitive, so widening read access has no cost.
- **Why no write policy.** The only legitimate writer is the seed migration, which runs as the migration role and bypasses RLS. The service-role client in `get_supabase_service_client` also bypasses RLS. Adding a write policy would create an attack surface for no benefit.
- **Forward-only.** No data migration; the existing 25 seed rows are unaffected.
- **Naming.** Policy name matches the casing/style of `"User badges are viewable by everyone"` in `045_user_badges.sql` for easy `grep`.
