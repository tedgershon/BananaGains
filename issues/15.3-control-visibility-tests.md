# 15.3 — UI control visibility tests (matrix §6)

## Scope

For every mutating control in matrix §6a (market detail page) and §6c (admin pages), assert visibility and enabled-state per role + context overlay. Plus the §6b global-nav controls.

This is the largest test sweep in doc 15. ~10 controls × ~5 role-contexts × edge cases = ~60–80 tests.

## Acceptance criteria

### File structure

- [ ] `frontend/e2e/controls/market-detail.spec.ts` — §6a (Place Bet, Propose Resolution, Submit Dispute, Cast Vote, Backroll, etc.).
- [ ] `frontend/e2e/controls/global-nav.spec.ts` — §6b (Sign In button, Profile dropdown, Notification bell, Admin nav link, Role-preview toggle).
- [ ] `frontend/e2e/controls/admin-pages.spec.ts` — §6c (Approve/Deny, Search Users, Promote/Demote, dashboard cards including "Admin Review" gap-B-4 test.fixme'd).

### Test pattern

For each control, parameterize across role-context tuples:

```ts
const placeBetCases = [
  { role: "anon", expected: "hidden" },
  { role: "user", marketStatus: "open", expected: "enabled" },
  { role: "user", marketStatus: "closed", expected: "disabled" },
  { role: "user", overlay: "creator", expected: "hidden" },
  { role: "user", overlay: "bettor", marketStatus: "open", expected: "enabled" },
  { role: "admin", marketStatus: "open", expected: "enabled" },
  { role: "super_admin", marketStatus: "open", expected: "enabled" },
];

for (const c of placeBetCases) {
  test(`Place Bet button — ${c.role}${c.overlay ? `+${c.overlay}` : ""} on ${c.marketStatus ?? "any"} market → ${c.expected}`, async ({ page, signInAs, seedMarket, seedBet }) => {
    const m = await seedMarket({ status: c.marketStatus ?? "open", creatorRole: c.overlay === "creator" ? c.role : "admin" });
    if (c.overlay === "bettor") await seedBet(m, { role: c.role, side: "yes" });
    if (c.role !== "anon") await signInAs(c.role);

    await page.goto(`/markets/${m.id}`);
    const button = page.getByRole("button", { name: "Place Bet" });

    if (c.expected === "hidden") await expect(button).not.toBeVisible();
    else if (c.expected === "disabled") await expect(button).toBeDisabled();
    else await expect(button).toBeEnabled();
  });
}
```

### Coverage requirement

- [ ] Every row in §6a, §6b, §6c has at least one test per non-trivial cell. "Trivial" means `hidden` for the same reason as another row already covered (e.g., `anon` is `hidden` for everything mutating — one anon test per page is enough, not one per control).
- [ ] **Gap B-2 verification** lives here too: a test that signs in as user A, has user B propose a market, then visits `/portfolio` as user A and asserts user B's market does NOT appear in the "Proposed Markets" section. Cross-references the §3b fix from doc 13.
- [ ] **Defense-in-depth assertion** (§6d): for each `hidden` cell on a mutating control, a paired test asserts the API call would also fail. Skip if doc 14.2 already covers it; otherwise include here as a sanity check (one or two examples is enough — doc 14 is the primary).

## Notes

- **Selecting controls reliably** — prefer `getByRole(..., {name})` over CSS selectors. If a control isn't selectable by role/name, that's an accessibility gap worth flagging in the PR (could spawn a separate small fix).
- **Overlay setup** — `seedMarket({creatorRole: "user"})` makes the test user the creator. `seedBet(m, {role: "user"})` makes the test user a bettor. The fixtures in 15.1 must support these.
- **Disabled state checking** — Playwright's `toBeDisabled()` checks the `disabled` attribute. If a button is "disabled-looking" via CSS class only (not the `disabled` attribute), this test won't catch it. Treat that as a UX bug worth fixing.
- **`/admin/admin-review` controls** are `test.fixme(...)`'d until Gap B-4 ships.
- The ~60–80 tests will take a few minutes locally. Acceptable; CI can parallelize.
