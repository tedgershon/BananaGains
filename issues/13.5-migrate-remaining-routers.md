# 13.5 — Migrate remaining routers + audit fixes for notifications/resolution/rewards

## Scope

Bundle migration for the seven smaller routers. Same `AppError` pattern as 13.3/13.4. Includes three audit fixes that are tightly coupled to the migration:

- **`notifications.py`**: per matrix §7f, non-owner access to `POST /api/notifications/{id}/read` and `DELETE /api/notifications/{id}` returns **404** (`NOTIFICATION_NOT_FOUND`), not 403. The current behavior may be 403 or may filter by user; standardize on 404.
- **`resolution.py`**: add `Depends(get_current_user_optional)` to `list_community_votes` (currently has no auth dependency) so doc 12's `set_user_context` fires uniformly.
- **`rewards.py`**: add `Depends(get_current_user_optional)` to `get_user_badges` (same reason). Route stays public per matrix §7h.

## Acceptance criteria

- [ ] For every file in scope: every `raise HTTPException(...)` replaced with `AppError(...)` using the appropriate `ErrorCode`.
- [ ] No `f"...: {e}"` interpolations in any of these files.
- [ ] `notifications.py`: confirm `POST /api/notifications/{id}/read` and `DELETE /api/notifications/{id}` return `AppError(404, ErrorCode.NOTIFICATION_NOT_FOUND, "Notification not found.")` when the notification exists but is not owned by `current_user`. **Per matrix §7f** — this is to prevent existence-probing.
- [ ] `resolution.py::list_community_votes` gains `_current_user: dict | None = Depends(get_current_user_optional)` parameter (unused in body — purpose is Sentry attribution).
- [ ] `rewards.py::get_user_badges` gains the same.
- [ ] Existing tests pass.

## Notes

- `bets.py` is the cleanest existing pattern (already maps DB errors to user messages without leaks). The migration there is mostly mechanical: `HTTPException(409, detail="Insufficient banana balance")` → `AppError(409, ErrorCode.BET_INSUFFICIENT_BALANCE, "Insufficient banana balance.", original=exc)`.
- The 404-vs-403 audit fix is the only real behavior change in this PR. Everything else is shape/contract changes.
- If a route currently returns the notification body when the user *is* the owner, leave that alone — only the non-owner branch changes.
