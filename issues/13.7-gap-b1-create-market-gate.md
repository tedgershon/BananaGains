# 13.7 — Gap B-1: server-side gate for `/markets/create` (and other matrix-§5 routes)

## Scope

Close matrix Gap B-1: `/markets/create` currently renders the market-creation form for unauthenticated users. The API call would fail with 401, but the UX is wrong and the form shape leaks. **Fix: server-side redirect before the page renders.**

Two valid approaches; pick **one** based on whether `frontend/src/middleware.ts` already exists:

- **Approach 1 — Server component with `redirect()`** at `/markets/create/page.tsx`. Local fix. Use this if no `middleware.ts` exists yet *and* you don't want to introduce one in this PR.
- **Approach 2 — Next.js middleware** at `frontend/src/middleware.ts`. Centralizes gating for all matrix §5 routes that say `redirect→/auth`. Use this if you want defense in depth across multiple routes in one PR.

**Recommended: Approach 2** if no middleware exists, since it also closes any other route that turns out to be only client-side-gated (`/portfolio`, `/profile`, `/rewards`).

## Acceptance criteria

### Approach 1 (route-local)

- [ ] `frontend/src/app/markets/create/page.tsx` becomes (or wraps) a server component that:
  ```tsx
  import { redirect } from "next/navigation";
  import { createSupabaseServerClient } from "@/lib/supabase-server";

  export default async function CreateMarketPage() {
    const supabase = createSupabaseServerClient();
    const { data: { session } } = await supabase.auth.getSession();
    if (!session) redirect("/auth");
    return <CreateMarketClient />;
  }
  ```
- [ ] The existing client-side form code is preserved as `CreateMarketClient` (extract if needed).

### Approach 2 (middleware)

- [ ] `frontend/src/middleware.ts` exists and exports `middleware(request)` + a `config` object with `matcher` covering: `/markets/create`, `/portfolio`, `/profile`, `/rewards`, `/admin/:path*`.
- [ ] Middleware reads the Supabase session via `createSupabaseMiddlewareClient`. If no session, returns `NextResponse.redirect(new URL("/auth", request.url))`.
- [ ] For `/admin/*`: also redirect to `/` (not `/auth`) if the session exists but the role is `user`. This matches matrix §5 ("admin redirects to `/`, not 403").

### Either approach

- [ ] Update matrix §8: delete the row for Gap B-1.
- [ ] Update matrix Changelog: "Gap B-1 fixed via PR-13.7 (approach N)."

## Notes

- **Defense in depth still required:** the API gate (`POST /api/markets` requires `get_current_user`) is already correct. This PR adds the page-rendering gate; both layers should be in place.
- If you choose Approach 2, you're effectively closing parts of any other matrix-§5 misalignment in the same PR. That's a feature, not a bug. List the routes you covered in the PR description so reviewers know.
- Do **not** use a client-side `useEffect` redirect. That ships protected component code to the browser and shows a flash before redirect — it's the failure mode this PR exists to fix.
- If the existing app already has a server-side helper for "get session in a server component," prefer it over re-inventing. Check `frontend/src/lib/supabase-server.ts` (if it exists).
