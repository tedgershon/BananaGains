# 12.4 — Frontend: `@sentry/nextjs` install, wizard config, `SessionProvider` user tag

## Scope

Install the Sentry Next.js SDK, run the wizard to scaffold the three runtime configs (client, server, edge), wrap `next.config.ts` with `withSentryConfig`, and tag Sentry with the authenticated user's UUID in `SessionProvider.tsx`. Source maps upload automatically on every Vercel deploy via the integration installed in 12.1.

This is one PR (not split further) because the wizard generates the four files together and the SDK init only makes sense once they're all in place.

## Acceptance criteria

### Install & wizard

- [ ] `pnpm add @sentry/nextjs` — adds `"@sentry/nextjs": "^9.0.0"` to `frontend/package.json`.
- [ ] Run `npx @sentry/wizard@latest -i nextjs` once; commit the generated diff. The wizard creates:
  - `frontend/sentry.client.config.ts`
  - `frontend/sentry.server.config.ts`
  - `frontend/sentry.edge.config.ts`
  - `frontend/instrumentation.ts`
- [ ] Wizard also wraps `frontend/next.config.ts` with `withSentryConfig(nextConfig, {...})`. Review the diff and confirm.

### Config tuning (post-wizard edits)

In all three `sentry.*.config.ts` files:

- [ ] `dsn` reads from the appropriate env var (`process.env.NEXT_PUBLIC_SENTRY_DSN` for client; `process.env.SENTRY_DSN` for server/edge).
- [ ] `environment` reads from `process.env.NEXT_PUBLIC_SENTRY_ENVIRONMENT` (client) or `process.env.SENTRY_ENVIRONMENT` (server/edge), defaulting to `"development"`.
- [ ] `release` reads from the corresponding env var (Sentry SDK reads `VERCEL_GIT_COMMIT_SHA` automatically when not set explicitly).
- [ ] `tracesSampleRate: 0.1`.
- [ ] `sendDefaultPii: false`.
- [ ] **Client only:** `replaysSessionSampleRate: 0`, `replaysOnErrorSampleRate: 1.0`. Add a `beforeSend` that strips any `andrew_id` field from `event.user`.

In `frontend/next.config.ts`:

- [ ] `withSentryConfig(nextConfig, { org: "bananagains", project: "bananagains-frontend", silent: !process.env.CI, widenClientFileUpload: true, hideSourceMaps: true, disableLogger: true })`.

### Auth tag in `SessionProvider.tsx`

- [ ] Import `* as Sentry from "@sentry/nextjs"`.
- [ ] Inside the existing `onAuthStateChange` callback, after the existing `setIsDemo(...)` logic:
  - If a session exists: `Sentry.setUser({ id: session.user.id })` — **UUID only**, never `andrew_id` or email.
  - If no session: `Sentry.setUser(null)`.

### Env vars (Vercel)

- [ ] Vercel project Settings → Environment Variables (Production scope) has all four populated (placeholders set in 12.1; this issue confirms they're correct):
  - `NEXT_PUBLIC_SENTRY_DSN`
  - `NEXT_PUBLIC_SENTRY_ENVIRONMENT=production`
  - `SENTRY_DSN` (same value as `NEXT_PUBLIC_SENTRY_DSN` for SSR/server-component capture)
  - `SENTRY_ENVIRONMENT=production`

## Verification

```bash
# Local dev: app boots without Sentry
cd frontend
pnpm dev
# Browser DevTools console: window.Sentry is undefined OR initialized with empty DSN no-op
```

```bash
# Build with Sentry env vars set
cd frontend
NEXT_PUBLIC_SENTRY_DSN=<dev DSN> NEXT_PUBLIC_SENTRY_ENVIRONMENT=development \
  SENTRY_DSN=<dev DSN> SENTRY_ENVIRONMENT=development \
  pnpm build
# Build log shows the Sentry source-map upload step succeeding
```

In a deployed environment:

- Throw a manual error from a button-click handler. Confirm the event appears in `bananagains-frontend` within 30s.
- The stack trace is **fully demangled** (component names visible, not `chunk-abc.js:1:42`).
- After signing in, the next captured event has `user.id` set to the Supabase UUID. After signing out, `user.id` is unset on the next event.
- Manually attach `Sentry.setUser({ id, andrew_id: "test" })` and trigger an event — confirm the captured event has **no** `andrew_id` (proves the `beforeSend` scrub).

## Notes

- The wizard occasionally injects extra config fields (e.g., `tunnelRoute`); review the diff and reject any not in the design doc.
- `widenClientFileUpload: true` is required because Next.js 16 splits client bundles into more chunks than the Sentry SDK's default upload pattern catches.
- `hideSourceMaps: true` removes `.map` files from the public bundle (they go to Sentry, not the user's browser).
